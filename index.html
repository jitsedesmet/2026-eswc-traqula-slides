SPARQL is not one language

  • Standard versions: SPARQL 1.0 → 1.1 → 1.2 (in progress)
  • Vendor extensions: Virtuoso (full-text search), Jena (quad construction), Oxigraph (ADJUST)
  • Vendor restrictions: some endpoints drop expensive operators (e.g. OPTIONAL)
  • W3C entering maintenance mode → faster version evolution ahead

Federated queries meet dialect diversity

Three key challenges

  1. Query evaluation — a SPARQL 1.2 query may fail on older endpoints
  2. Tooling — linters, editors, and formatters assume a single SPARQL version
  3. Maintainability — multi-version support leads to messy, hard-to-maintain code

Existing parsers don't cover all needs

13 of 16 popular parsers use generated grammars (ANTLR, Bison, JavaCC…)

  • No rule-level modularity
  • No round-tripping
  • Not web-native
  • Language-specific compilation step

None satisfies
all four needs

Introducing Traqula

  • Modular SPARQL toolkit
  • 🔍 Parsing — query string → AST or algebra
  • ✏️ Generation — AST or algebra → query string
  • 🔄 Transformation — rewrite queries between dialects
  • TypeScript · runs in browsers and Node.js

Three query representations

Plug-and-play grammar rules

Each rule is a named module:

  • Add a rule for a new language feature
  • Remove a rule to restrict the language
  • Patch a rule to change its behavior

Assemble parsers, generators,
and transformers from these modules.

// Base IRI generator
const base = GeneratorBuilder
  .create([iriFull, prefixedIri])
  .addRule(iri);

// Patch it for a custom dialect
const custom = GeneratorBuilder
  .create(base)
  .patchRule(altPrefixedRule)
  .build();

Round-tripping: change only what you intend

Parse → transform at AST level → regenerate = only intentional changes

Before
SelECT * { ?s   ?p ?o }
→
Rename ?s → ?subject
SelECT * { ?subject   ?p ?o }

Spacing, casing, and punctuation are preserved

Enables: linters · formatters · refactoring tools · language servers
